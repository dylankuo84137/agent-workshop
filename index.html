第一章 · 命名你的痛

兩個痛點，其實是同一個問題

power user 撞的牆不是「不會用」，而是這兩道——而它們是同一條原則解掉的。

approval friction

點頭疲勞

它一直回頭問你：「可以讀這個檔嗎？」「可以跑這個指令嗎？」每一個動作都要你點頭。 你的角色從「下決策的人」退化成「一直按同意的人」。

tracking friction

盯場疲勞

你得盯著它一步步做完，一次一件事。想同時開三件？開了也是空轉，因為每一件都在等你回去點頭。 你被綁在螢幕前，走不開。

🏫 現場你叫 agent 把這學期 12 個孩子的觀察紀錄整理成一份學期報告。 它讀第 1 個檔——「可以讀這個檔嗎？」你按同意；讀第 2 個檔又問……問到第 12 次你開始煩（點頭疲勞）。 你乾脆想同時叫它把班網的教學資源也更新了，卻發現得先盯完報告這件、才騰得出手開下一件（盯場疲勞）。 你買了個助理，結果一整個下午都黏在螢幕前幫它點頭、替它排隊。

這不是 AI 不夠聰明，是把關的位置放錯了——你在「每一個輸入」上把關， 所以它每一步都得停下來等你。

第二章 · 一個原則

把關卡，從「輸入」搬到「輸出」

在 sandbox（隔離的工作區）裡讓它無須看顧地自己做；只在閘門（Responsibility Gate）——那些不可逆、且後果由你扛的邊界——把關一次。

牆內 · 工作區

Sandbox 內 — 無須看顧

  • 讀、寫、改、搬工作區內的檔案
  • 跑指令、試錯、重來
  • 不問你，一路做到底

最壞的情況：把工作區重置，重來一次。

閘門把關一次
牆外 · 不可逆

邊界 — 停下來問

  • push / 發布 / 部署
  • 寄信、送出、貼到對外服務
  • 刪除或覆蓋工作區以外的東西
  • 花錢、難以復原的動作

共同點：不可逆，而且責任在你、不在它——後果由你扛，就由你按下去。

一個動作，同時解兩個痛：它在 sandbox 內不問你了（解點頭疲勞），於是你也能走開 （解盯場疲勞）。你的角色也跟著換一格——從「每個動作都要你點頭」的 human-in-the-loop（人在環中），變成「只在閘門盯一下」；這是區分代理的自主程度。而它在 sandbox 內「你不在場、自己把事做完」的這份能力，業界叫 agentic autonomy（代理式自主）——焦點在「它得到的能力」，不是「你放掉了什麼」。至於 sandbox，是你工具裡看得到、業界也通用的詞（Codex 的 sandbox mode、Claude Code 的 worktree／/sandbox）——這些我們直接用業界常用的詞。

🏫 現場同一份學期報告，換成「無須看顧」的姿態：你說一聲方向，它在工作區裡 12 個檔一路讀完、整理、產出草稿，中間完全不回頭問你。等草稿放到桌上，你才看一眼（這就是 validate）； 真正要「寄給家長」那一步——那才是閘門——它才停下來等你按下送出。沙箱內的試錯再多都沒關係—— 最壞就是把工作區重置、重跑一次；會痛的只有「寄出去」那一下，而那一下你還握在手上。

🌙 白班規劃 ＋ 夜班執行

把工作想成兩班制：白天你規劃（定方向、切任務），晚上它執行 （在 sandbox 內自己跑）。你不用守夜，早上來閘門驗收。

第三章 · 紀律

中間敢不盯著，因為判斷在前、驗收在後

砍掉「每一步都點頭」之後，會留下一個洞：「我憑什麼敢不盯著它？」 答案是把判斷前置、把檢查後置，中間才敢整段交給它。這就是 PIV。

PIV 三拍白話注意力在誰身上
Plan
規劃		它先提案要做什麼、怎麼驗收；你確認方向人：定方向、切任務
Implement
執行		方向對了，它在 sandbox 內一路做完，不問你機器：不打擾你
Validate
驗收		在閘門看成果、看它怎麼驗證的人：只看這一關

兩種 PIV：你多半用「對話式的 Lightweight PIV」

PIV 分兩層：要拿 AI 管程式、改動回滾成本高的工作，用嚴格版（Strict PIV）——計畫、執行、驗收各自獨立 session，屬於後面的進階章節；你們日常的備課、改寫、整理，Lightweight PIV 就夠了。它不需要任何正式檔案、不需要 git，就是一段對話：它提案 → 你確認 → 它做 → 你在閘門看一眼。兩層共同的鐵則是「不未知先行」——沒得到你確認，它不做有後果的事。這套紀律已經寫進等一下要裝的「代理指示」裡，所有工具通用。

「你的注意力，是 validate 的訓練資料。」

盯到它做錯，不要只修那一個 bug——在驗收清單加一條 check， 讓「這一類錯」以後到不了閘門。於是盯場這件事會自我清算：盯一次，少一次。

🏫 現場第一次它把報告裡的專有名詞翻成了中文。你不要只改那一個詞—— 在「代理指示」加一句 “proper nouns stay in English”。下一次、下下次，它都不會再犯。 你每一次的盯場，都是在餵 validate 學會自己把關；盯著盯著，你就能離開螢幕了。
第四章 · 認識你的工具

要善用工具，先知道工具底下有什麼

使用代理卡住，常常不是不會用 AI，是不知道工具底下有一層「技能／規範」存在。

🏫 現場在座對「技能」的熟悉度不一樣：有人已經自己建過 skill，有人還沒搞懂 skill 是什麼——都正常。 其實你們早就在用這層了：TeacherOS 底下那些技能，幫你們用 git、產教案、部署資源。今天不是教你用 AI（你已經會了）， 是把這層蓋子打開，讓還沒建過的人也看懂、會改，已經會的人知道往哪裡擴。

先講清楚：什麼是 skill？

一個資料夾 ＋ 一個 SKILL.md 檔，就這樣。檔案開頭用一句話描述「什麼時候該用我」（觸發語／description）， 下面是給 AI 讀的指示（內文）。當你講的話對上那句觸發語，AI 就自動把這份指示讀進來照做。 說穿了，skill 就是把一段你反覆在用的工作流程封裝起來——存一次，以後一句話就喚出來，AI 每次都照同一套做。

Windows 11 · 動手前的環境

給 Windows 同事：先把環境接好

好消息：這四個工具現在都能在 Windows 上跑。 差別在「沙箱自主」這件事——CLI 工具（Claude Code／Codex）走 WSL 最穩、路徑也跟本頁一致； GUI 工具（Cowork／Antigravity）是原生 app，根本不碰 WSL。先看你屬於哪一種。

Windows 怎麼裝、怎麼用：

  • 下載安裝：VS Code 到 code.visualstudio.com（Windows 選 User Installer——免系統管理員權限、更新較順，官方也推薦新手用這個）；Cursor 到 cursor.com（舊網址 cursor.sh 已轉址，認 .com）。下載到的安裝程式雙擊執行即可。
  • 開啟一個資料夾：選單 File → Open Folder，挑你的專案／教學資料夾，左邊就會列出裡面的檔案。
  • （可選）用指令開：裝完重開終端機，就能在任何資料夾打 code .（VS Code）或 cursor .（Cursor）直接開它。Cursor 的 cursor 指令要先在編輯器裡按 Ctrl+Shift+P → 執行「Shell Command: Install 'cursor' command in PATH」一次。

走 WSL 路線的人請注意：VS Code 與 Cursor「連進 WSL」的方式不一樣（Cursor 不能用微軟的 WSL 擴充），做法看下面〈③ 串接你現有的 Windows 工具〉。

想讀／編 Markdown？裝 Obsidian

你的 AI 會產出一堆 Markdown——交接文件、ADR、CONTEXT.md、各種 .md 筆記。用編輯器看是「看原始碼」； 想舒服地讀（標題、清單、callout 都漂亮排好，還能在筆記之間跳來跳去），就用 Obsidian。 它的核心概念很簡單：把一個資料夾當成「vault」打開，裡面的 .md 就是你的筆記。

  • 下載安裝：到 obsidian.md（只認官網，搜尋結果常混山寨站），下載 Windows 安裝程式執行。個人／教育使用完全免費，免註冊。
  • 開一個既有的資料夾：開 Obsidian → 歡迎畫面點 Open folder as vault（或左下角切換 vault 圖示 → Manage Vaults… → Open folder as vault）→ 選你那個裝著 .md 的資料夾。它只會多放一個隱藏的 .obsidian 設定夾，不會動到你原本的 .md。
  • 讀就對了：標準 Markdown、callout、frontmatter（YAML properties）、[[wikilink]] 都會漂亮 render，對讀 AI 產生的文件綽綽有餘。

小提醒：AI 產生的文件裡多半是一般 Markdown 連結／檔案路徑，不是 Obsidian 專屬的 [[wikilink]]，所以「點了不會跳」是正常的、不是壞掉。

WSL 使用者注意：別把 Obsidian 直接指向 \\wsl.localhost\… 的 WSL 路徑。 Obsidian 是原生 Windows app，開 WSL 網路路徑會明顯變慢、檔案監看（自動偵測外部變更）不可靠、還常直接報錯（EISDIR）。 務實做法：要常用 Obsidian 看的筆記，放在 Windows 這一側（例如 C:\Users\你\Notes）； 跑在 WSL 裡的 agent 要存取時，從 WSL 端走 /mnt/c/Users/你/Notes 反向讀寫即可。

① 安裝 WSL（要用 Claude Code，或想讓 Codex 跑在 Linux 的人）

  • 用系統管理員開終端機：開始選單找 Windows Terminal（Windows 11 內建）或 PowerShell，右鍵「以系統管理員身分執行」。
  • 跑一行安裝指令：輸入下面那行，它會自動裝好 WSL 2 ＋ Ubuntu。
  • 重開機：裝完需要重新啟動電腦。
  • 設 Linux 帳號：重開後從開始選單開 Ubuntu，設一組使用者名稱與密碼（打密碼時畫面不會動是正常的；這組帳號跟你的 Windows 帳號無關）。
  • 確認成功：跑 wsl -l -v，看到 Ubuntu、VERSION 顯示 2 就好了。
⌨️ PowerShell（以系統管理員身分執行） 
# 一行裝好 WSL 2 + Ubuntu，跑完重開機
wsl --install

# （重開機後，第一次開 Ubuntu 會要你設 Linux 使用者名稱／密碼）

# 確認裝好了、而且是 WSL 2
wsl -l -v

需求：Windows 11（或 Windows 10 build 19041+）、x64 或 Arm64。wsl --install 會自動把需要的 Virtual Machine Platform 一起開好，不用手動到 BIOS 設定。

② 進 WSL、把 CLI 裝進去

每天怎麼進 WSL：開 Windows Terminal，按分頁列上的 ∨ 箭頭 → 選 Ubuntu（或在 PowerShell 裡打 wsl）。你就進到 Linux 了。

在 WSL 裡裝工具（用 Linux 指令，不是在 PowerShell）：

🐧 在 WSL（Ubuntu）終端機裡 
# Claude Code：在 WSL 內安裝，也在 WSL 內啟動
curl -fsSL https://claude.ai/install.sh | bash

# 之後 cd 到你 WSL 家目錄裡的專案資料夾，啟動
claude

關鍵：CLI 要在 WSL 終端機裡安裝、在 WSL 裡啟動，不要從 PowerShell 跑——那會變成另一套原生 Windows 安裝（設定也存在不同地方）。

③ 串接你現有的 Windows 工具

WSL 不是要你換掉熟悉的介面——它就接在你本來就在用的 VS Code 和檔案總管底下。

VS Code
照常用 Windows 上的 VS Code，裝一個 WSL 擴充套件（Extensions 搜 WSL，發行者 Microsoft，舊名 Remote - WSL）。 然後在 WSL 終端機裡打 code .——VS Code 視窗開在 Windows，但終端機、擴充套件、Claude Code 全都在 WSL 裡跑。 你看到的是熟悉的 Windows 視窗，底下是 Linux。
Cursor
Cursor 雖是 VS Code 的 fork，卻不能用微軟那顆 WSL 擴充（微軟把 Remote 系列鎖在自家正版產品上）。 它自帶一套 WSL 連線（Anysphere 的 anysphere.remote-wsl，內建、免另外找）——一樣在 WSL 終端機裡打 cursor .， 或點視窗左下角的遠端圖示 → Connect to WSL。別照舊教學去裝微軟的 Remote-WSL，那顆在 Cursor 裝不起來。
檔案總管
從 Windows 看 WSL 檔案：檔案總管網址列貼 \\wsl.localhost\Ubuntu\home\<你的名字>\（舊寫法 \\wsl$\Ubuntu\… 也通）。
從 WSL 看 Windows 檔案：/mnt/c/Users/<你的名字>/。

一條鐵則：專案／教學檔放在 WSL 家目錄（~/）裡，不要放在 /mnt/c。 跨系統存取（Windows ↔ Linux）官方明講會明顯變慢。把檔案留在 Linux 這一側，工具跑起來才順。

不裝 WSL？原生 Windows 路徑對照

用 Cowork／Antigravity（或 Codex 原生）的人不需要 WSL。本頁所有 ~/ 開頭的路徑，原生 Windows 這樣換：

本頁寫法（WSL／macOS／Linux）原生 Windows 換成
~/%USERPROFILE%\（即 C:\Users\<名字>\）
路徑分隔 /\
~/.claude/skills/<名稱>/SKILL.md%USERPROFILE%\.claude\skills\<名稱>\SKILL.md

小提醒：原生 Windows 與 WSL 的設定是兩份、各自獨立——~/.claude（WSL 裡）和 %USERPROFILE%\.claude（Windows 裡）是不同位置、不會互通。挑一邊用，別混。最後〈一鍵設定〉那段貼給 AI 時，它也會自動幫你把路徑換成你那一邊。

## Domain awareness

During codebase exploration, also look for existing documentation (CONTEXT.md at the root, ADRs under docs/adr/). Create these files lazily — only when you have something to write. If no CONTEXT.md exists, create one when the first term is resolved.

## During the session

### Challenge against the glossary
When the user uses a term that conflicts with the existing language in CONTEXT.md, call it out immediately.

### Sharpen fuzzy language
When the user uses vague or overloaded terms, propose a precise canonical term.

### Discuss concrete scenarios
Stress-test domain relationships with specific scenarios that probe edge cases and force the user to be precise about boundaries between concepts.

### Cross-reference with code
When the user states how something works, check whether the code agrees. Surface contradictions.

### Update CONTEXT.md inline
When a term is resolved, update CONTEXT.md right there — don't batch. CONTEXT.md is a glossary and nothing else: totally devoid of implementation details. Use the format in CONTEXT-FORMAT.md.

Only offer an ADR when all three are true: (1) hard to reverse, (2) surprising without context, (3) the result of a real trade-off. Use the format in ADR-FORMAT.md.

# CONTEXT.md Format

## Structure

# {Context Name}

{One or two sentence description of what this context is and why it exists.}

## Language

**Order**:
{A one or two sentence description of the term}
_Avoid_: Purchase, transaction

**Invoice**:
A request for payment sent to a customer after delivery.
_Avoid_: Bill, payment request

**Customer**:
A person or organization that places orders.
_Avoid_: Client, buyer, account

## Rules

- **Be opinionated.** When multiple words exist for the same concept, pick the best one and list the others under `_Avoid_`.
- **Keep definitions tight.** One or two sentences max. Define what it IS, not what it does.
- **Only include terms specific to this project's context.** General programming concepts (timeouts, error types, utility patterns) don't belong even if the project uses them extensively. Before adding a term, ask: is this a concept unique to this context, or a general programming concept? Only the former belongs.
- **Group terms under subheadings** when natural clusters emerge. If all terms belong to a single cohesive area, a flat list is fine.

## Single vs multi-context repos

**Single context (most repos):** One CONTEXT.md at the repo root.

**Multiple contexts:** A CONTEXT-MAP.md at the repo root lists the contexts, where they live, and how they relate to each other. If a CONTEXT-MAP.md exists, read it to find contexts; if only a root CONTEXT.md exists, single context; if neither exists, create a root CONTEXT.md lazily when the first term is resolved.
📎 grill-with-docs/ADR-FORMAT.md 
# ADR Format

ADRs live in `docs/adr/` and use sequential numbering: `0001-slug.md`, `0002-slug.md`, etc.

Create the `docs/adr/` directory lazily — only when the first ADR is needed.

## Template

# {Short title of the decision}

{1-3 sentences: what's the context, what did we decide, and why.}

That's it. An ADR can be a single paragraph. The value is in recording *that* a decision was made and *why* — not in filling out sections.

## Optional sections

Only include these when they add genuine value. Most ADRs won't need them.

- **Status** frontmatter (proposed | accepted | deprecated | superseded by ADR-NNNN) — useful when decisions are revisited
- **Considered Options** — only when the rejected alternatives are worth remembering
- **Consequences** — only when non-obvious downstream effects need to be called out

## Numbering

Scan `docs/adr/` for the highest existing number and increment by one.

## When to offer an ADR

All three of these must be true:

1. **Hard to reverse** — the cost of changing your mind later is meaningful
2. **Surprising without context** — a future reader will look at the code and wonder "why on earth did they do it this way?"
3. **The result of a real trade-off** — there were genuine alternatives and you picked one for specific reasons

If a decision is easy to reverse, skip it. If it's not surprising, nobody will wonder why. If there was no real alternative, there's nothing to record.

# To Issues

Break a plan into independently-grabbable issues using vertical slices (tracer bullets).

The issue tracker and triage label vocabulary should have been provided to you — run /setup-matt-pocock-skills if not.

## Process

### 1. Gather context
Work from whatever is already in the conversation context. If the user passes an issue reference (number, URL, or path), fetch it and read its full body and comments.

### 2. Explore the codebase (optional)
Understand the current state of the code. Issue titles and descriptions should use the project's domain glossary, and respect ADRs in the area you're touching.

### 3. Draft vertical slices
Each issue is a thin vertical slice that cuts through ALL integration layers end-to-end, NOT a horizontal slice of one layer.
- Each slice delivers a narrow but COMPLETE path through every layer (schema, API, UI, tests).
- A completed slice is demoable or verifiable on its own.
- Prefer many thin slices over few thick ones.
Slices may be 'HITL' (needs human interaction, e.g. an architectural decision) or 'AFK' (can be implemented and merged without human interaction). Prefer AFK where possible.

### 4. Quiz the user
Present the proposed breakdown as a numbered list. For each slice show: Title, Type (HITL/AFK), Blocked by, and user stories covered. Ask whether the granularity, dependencies, and HITL/AFK marks are right. Iterate until the user approves.

### 5. Publish the issues
For each approved slice, publish a new issue using this body template, in dependency order (blockers first):

## What to build
A concise description of this vertical slice. Describe the end-to-end behavior, not layer-by-layer implementation. Avoid specific file paths or code snippets — they go stale fast.

## Acceptance criteria
- [ ] Criterion 1
- [ ] Criterion 2

## Blocked by
- A reference to the blocking ticket (if any), or "None - can start immediately".

Do NOT close or modify any parent issue.

設定你的 agentic autonomy

三件事：①放一份代理指示（告訴 AI 我們的工作姿態），②打開自動放行＋設好閘門把關，③決定要不要、以及怎麼讓它連網。 選你的工具分頁，照著做。

共用：代理指示（agent instructions）的內文

這類常駐指示在 AI agent 圈通稱 agent instructions（代理指示）——放進 AGENTS.md／CLAUDE.md 這類檔案， 定義 agent 該怎麼動；我們這份特別聚焦在「sandbox 內無須看顧、邊界停下來問」。下面內文四個工具通用，只是放的位置與檔名不同（分頁裡會說）。 保持英文，因為這是給 AI 讀的。

為什麼這個檔案，值得你最先花時間？

它是 AI 每一個 session 一開場就讀進去的常駐合約——不像你某次對話裡臨時丟的一句指示（關掉就忘）， 它永遠在場，定義 agent 怎麼想、怎麼做、怎麼自我驗收。所以它是你整套設定裡槓桿最高的一個檔： 改一行，之後每一次對話都照新規矩跑。代價是它也最該保持精簡——它吃掉的是每個 session 開場的注意力預算， 塞太多，模型一開始就先被自己的規範拖累（就是第七章說的「context 有上限、會越用越笨」）。一份好的代理指示：短，只放真正改變行為的規則。

⚠️ 別覆蓋掉你「原本就有」的 CLAUDE.md／AGENTS.md

很多人早就有一份了，就放在你的專案根目錄。下面這份代理指示是要加進去、不是整檔換掉： 把它當成在現有內容後面新增一個段落（或小心合併），千萬別讓 AI 用它覆寫整個檔—— 否則你原本的稱呼、語言、寫作風格、專案規則會一起被清掉。 等一下〈一鍵設定〉給 AI 的指令裡，已經幫你寫「先檢查檔案存不存在，有就附加、不覆蓋」這條。

# Agent Instructions — Sandbox Autonomy with Boundary Approval

You are my hands-on agent. We work in a sandbox (an isolated workspace).
Inside the sandbox, act. At the boundary, pause. The point is to move the
checkpoint from the INPUT (every tool call) to the OUTPUT (what crosses an
irreversible boundary).

## Inside the sandbox — act, don't ask
- Read, write, edit, move, and delete files within this workspace freely.
  Do not ask for step-by-step approval. The worst case inside the sandbox is
  recoverable: we can reset the workspace.
- Run as many steps as the task needs without checking back. Batch the work
  and report once at the end, not after every action.

## At the boundary — stop and get my explicit OK
Treat these as irreversible. Always pause and ask before:
- Pushing, publishing, or deploying (git push, releasing, going live)
- Sending anything to a person or service (email, message, posting to an API)
- Deleting or overwriting anything OUTSIDE this workspace
- Spending money, or any action that is hard to undo

## How we work — Lightweight PIV
For any non-trivial task:
1. Plan — Briefly propose what you'll do and how you'll check it worked.
   State assumptions; if several interpretations are reasonable, list them
   instead of silently picking one.
2. Implement — Once I confirm the direction, do the whole thing in the
   sandbox without stopping to ask.
3. Validate — At the door, show me the result and how you verified it.
   I review the OUTPUT, not every step.

## Conventions — behavioral guardrails for every task
These bias toward caution over speed; for a trivial task, use judgment.
- Think before acting. State your assumptions out loud. If you're unsure, ask.
  If a request could mean more than one thing, lay out the options instead of
  silently picking one. If something is unclear, stop and name what's confusing.
- Simplicity first. Do the minimum that actually solves what I asked. No
  speculative extras, no over-building, no solving problems I didn't raise.
- Surgical changes. Touch only what the task needs. Don't "improve", reword, or
  reformat things I didn't ask you to. Match the existing style. Every change
  should trace back to my request.
- Goal-driven execution. Turn each task into a checkable success criterion
  before you start — "tidy this up" becomes "every entry has a date and a tag,
  then show me." Clear criteria let you run on your own; vague ones make you
  keep interrupting me.
- Verification-led. Decide how you'll check the work before you do it. Never
  claim something works — that a tool behaves a certain way, that a file came
  out right — from reasoning alone. Actually run it or look, then tell me what
  you saw.

## My attention is your validation training data
When I catch a mistake at the door, don't just fix that one instance. Add a
check — a rule, a test, a step in your process — so that whole class of
mistake can't reach the door again. Every catch should mean one fewer thing
for me to catch next time.

## Keep your context lean — our context window is finite and you dull as it fills
- For research-heavy work — web lookups or reading lots of files — delegate it to a
  subagent if your tool supports one, and pull back only the summary. Don't flood our
  main conversation with raw search results or file dumps I won't reread; that clutter
  is what makes you lose the thread.
- When this conversation runs long and you feel yourself drifting — rereading things,
  repeating yourself, missing instructions I gave earlier — say so out loud and offer
  to write a handoff and continue in a fresh session. Don't quietly soldier on while
  degrading; a stale long thread is the dumber version of you.

## Communication
Be concise. Lead with the result. Sacrifice grammar for brevity when reporting.

把信任也「開」起來：ask → allow 的機制

第三章是「抓到錯就加一條 check」往下核准；這裡是反方向——某一類動作你連續放行 5 次、結果都沒被你改過，就把它從 ask 升上 allow（白名單）。5 次是保守起點，可自己調；而且是逐動作類型allow，不是整隻 agent 一次放掉。兩個機制一起轉：錯的越擋越多、信任的越放越寬，要你親自過目的事只會越來越少。

第三個開關：要不要讓它連網？

沙箱多半預設斷網或限制連網——這是安全設計：就算它讀到惡意網頁裡夾帶的指令（prompt injection），也沒有現成管道把你的資料送出去。 但實際做事一定會用到網路：查資料、抓文件、裝套件，尤其是派 subagent 上網查（第七章）。好消息是——四個工具多半是「它要連網時停下來問你、你按准就好」，不一定要預先全開。 一個提醒：連網＝資料能出去，本身就是第二章那道閘門（送到對外服務）。所以在自動放行下讓它讀不可信的外部網頁時要特別小心——夾帶的注入指令可能透過你已允許的網域把東西外送（這有真實案例）。各工具怎麼開，看下面分頁新增的「連網」列。

自動放行
UI 最快：Antigravity 用「自主程度預設檔」一鍵套——首次啟動的設定精靈、或日後 Settings → Advanced Settings 裡選一檔：Review-driven（每個關鍵點都問你，官方建議新手起手）→ Agent-assisted（部分版本標為 Secure）→ Agent-driven（幾乎全自動推進）→ Custom（逐項自訂）。要本頁這種姿態，選 Agent-driven 或 Custom。
背後的細項旋鈕（選 Custom 時可逐項調，也是上面預設檔在動的東西）：Terminal 指令自動執行 三檔 Off／Auto／Turbo（Turbo＝全自動跑、只擋 Deny）；審查政策 Request Review／Agent Decides／Always Proceed。邊界用 Unified Permission 的 allow/deny（判定順序 Deny > Ask > Allow）。（預設檔名稱與少數標籤各版本略有差異，以你機器上實際看到的為準。）
連網
分兩種、各自設：① 讀網頁／內建瀏覽器（read_url）預設是 Ask——它要上網就問你，按「always allow」或把網域加進瀏覽器 allowlist 即放行（denylist 一律優先，連不到檢查服務時預設擋）。② 終端機 curl 這類走 command 權限，在最高自動模式（你設的 Turbo）下會自動跑、除非列入 Deny。所以要讓它查網路，重點是放行 read_url。
閘門把關
Turbo 是「全放、只擋 Deny」，所以 Deny 清單要備齊：社群常見建議（非 Google 官方明列）至少放 rm、sudo、curl，再加上 push／發布／部署類動作。

第七章 · 第三種稀缺

context 是有限的——對話越長，它越笨

解掉兩個摩擦後，還有一個你一直在花、卻看不到的資源：模型當下的注意力（context window）。 它不只有限，還會邊填邊退化——塞過某個量，你的 agent 會肉眼可見地變笨。 第六章那份代理指示，要它「keep your context lean」是 AI 那一側；這一章是你這一側：學會看出「這段對話已經太長了」。

什麼是 Dumb Zone（變笨區）？

模型在一段對話的前段最清醒（約前 100k tokens 是它狀態最好的 Smart Zone）； 一旦 context 塞超過某個量（Opus 體感約 250k tokens，叫它 Dumb Zone），它開始失焦—— 漏掉你的需求、忘記你前面交代過的規矩、該用的技能也不用了。 一段又臭又長的對話，裝的就是「比較笨的那一版」你的 agent。 這時候再叫它驗收自己剛剛寫的東西，等於請一個已經累壞的人校對自己的初稿——它根本看不出自己的錯。

🏫 現場你跟 agent 來回弄那份12 個孩子的學期報告弄了一個多小時。 它突然又問你一次早就講好的格式、把你前面交代過的「專有名詞留英文」忘了、連你裝好的技能都沒觸發。 這不是它突然變笨——是這段對話滑進 Dumb Zone 了。該做的不是繼續跟它盧，是換一段乾淨的對話接著做。

怎麼察覺「快滿了」？——四個工具的 context 量表

麻煩的是：四個工具裡只有兩個會讓你看到用量。Claude Code 和 Codex 有真的量表可以盯； Cowork 和 Antigravity 幾乎是飛盲——所以這兩個你更要靠「行為線索」＋養成主動重開的習慣。

讓 AI 自己舉手：第二雙眼睛

你不必獨自盯著量表。第六章那份代理指示，我們已經加了一條： 當它感覺自己開始飄（一直重讀、重複、漏掉你前面講的）就主動講出來，並提議寫一份 handoff、換新對話繼續。 於是「察覺對話太長」變成你和 AI 一起看——尤其在那兩個沒量表的工具上，這條指示就是你的警報器。

快滿了，三個動作（看狀況挑一個）

動作什麼時候用怎麼做
handoff
接力換對話		 對話已經很長、但任務還沒做完，捨不得丟掉進度 用第五章裝好的 handoff 技能：把目前進度壓成一份交接文件，開新對話讀它接著做。乾淨的 context ＋ 不掉進度。
開新 session
沿時間切		 到了自然的階段交界（規劃完→要執行、執行完→要驗收） 每個階段各開一段新對話（/clear 或新 task），讓負責驗收的它保持清醒。詳見第八章 Strict PIV。
subagent
外包研究		 同一個任務裡要查一堆網路資料、讀整個資料夾、翻長文件 派一個 subagent 去做，它在自己的 context 燒 token，只把摘要帶回主對話（下面有各工具對照）。

還有一個常駐的省法不在這張表裡：把第六章那份代理指示寫精簡——它每個 session 一開場就吃掉一塊注意力預算，越短，留給真正工作的就越多。

subagent：把最燒 token 的那段外包出去

「開新 session」是沿時間切（規劃／執行／驗收各開新對話）；subagent 則是在同一個任務裡把最燒 token 的那段外包出去。 要查一堆網路資料、讀一整個資料夾、翻長文件時——別讓這些原始結果灌進主對話（那正是把模型推進 Dumb Zone 的元兇）。 改成：派一個 subagent 去做，它在自己的 context 裡燒 token，只把「結論／摘要」帶回來，主對話保持精簡、繼續清醒。要它上網查，記得先授權連網（第六章）。

你的工具主 agent 能不能派 subagent？怎麼用
Claude Code能 · 原生叫它「派一個 subagent 去查 X」即可；或把角色定義在 ~/.claude/agents/。內建的 Explore agent 本身就會上網查、只回摘要。
Codex能 · 預設開自然語言交辦，或用 /agent 管理；自訂放 ~/.codex/agents/。它會 spawn 平行子 agent 再彙整成一個回覆。
Antigravity能主 agent 用 invoke_subagent 開一個獨立 context 的子 agent 代跑。注意：這跟「Agent Manager 同時跑多隻 agent」（你自己開多隻）是兩回事。
Claude Cowork沒有可自訂的它只會自動把大工作內部拆小（官方叫 sub-agent coordination），你不能自己建／指定；而且整段是單一連續 thread。

心法：主對話只留決策與結論，把「翻找」丟給 subagent。這跟「代理指示要精簡」是同一件事的兩面——都在省你最稀缺的資源：模型開場與當下的注意力。

第八章 · 進階

Tier 2：PIV ——「斜線指令」底下其實也是 skill

看 Claude Code 的 ai-layer-template 就清楚：PIV 同時活在兩層—— 一支 piv-loop 的 SKILL.md（把「三段分離」這套方法寫成可安裝的技能）， 外加 /plan /implement /validate 三個斜線指令 ＋ 一道 hook 強制的驗證閘來執行它。 能跨工具帶走的是那支 skill（方法本身）；帶不走的是 hook 強制的 gate——別的工具裝得到紀律，裝不到「強制」。

  • 分離三個 session：規劃、執行、驗收不要擠在同一個對話。原因就是第七章那個 Dumb Zone——三件事擠在同一段長對話，到驗收時它早就滑進變笨區，看不出自己剛剛犯的錯。每階段開新對話，是讓負責驗收的它保持清醒。
  • 計畫檔當交接介面：規劃完把計畫寫成一個檔，開新對話讀它再執行。檔案是唯一可靠的記憶。
  • 先定驗收，再動手：「加驗證」要先變成「為錯誤輸入寫測試，再讓它通過」。強的驗收條件，才敢整段交給它。

非 Claude Code 使用者照樣拿得到「方法」——它就寫在你「代理指示」裡的 Lightweight PIV（對話版）； 想更完整可參考 piv-loop 那支 skill 的內容（裡面 /plan 等指令是 Claude Code 專屬，但「三段分離 ＋ 計畫檔交接」的方法通用）。 Dylan 會在現場用 Claude Code 示範 hook 強制的完整版（多 worktree 並行）。

📦 想看真實長怎樣？這份公開範本可以參考

這套「PIV ＋ hook 強制閘」的完整實作是公開的，可以 clone 下來對照研究： github.com/cixin-dev/ai-layer-template。 重點檔案：.claude/skills/piv-loop/SKILL.md（方法本身）、.claude/commands/ 裡的 plan／implement／validate（三個斜線指令）、 .claude/hooks/security_guard.py（PreToolUse：擋 force-push／推 main／遞迴刪除／curl｜bash／讀 .env）、 .claude/hooks/validate_gate.py（Stop：跑 validate.sh 當驗收閘）。安裝靠 scripts/sync.sh，會把 defaultMode:"auto" 寫進全域 ~/.claude/settings.json（正是上一章說的「要放全域」）。

想在同一個任務裡省 context（把最燒 token 的研究外包出去），看第七章的「subagent」那一節——這裡專講沿時間切的 session 分離。
